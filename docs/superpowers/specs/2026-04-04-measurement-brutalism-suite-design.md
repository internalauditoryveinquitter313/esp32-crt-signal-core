# Measurement Brutalism Suite Design

**Date:** 2026-04-04  
**Status:** Draft

## Goal

Define the first authored showcase for the ESP32 CRT Signal Core as a short signal-art suite that is also a repeatable measurement protocol.

The suite should do two jobs with the same runtime:

- act as a public-facing demonstration of the library's signal-first architecture
- produce deterministic states suitable for electrical measurement and host-side analysis

This milestone is not the general-purpose compositor. It is a narrowly scoped bridge between the current calibration-pattern runtime and the future PhosphorOS composer.

## Design Summary

The suite is a controlled performance built from the same pattern families already present in the repository:

- amplitude staircases
- line-pair and transition stress regions
- full and windowed ramps

The visual direction is **Measurement Brutalism**:

- sparse
- technical
- high-contrast
- anti-nostalgic
- explicit about signal behavior instead of hiding it

The runtime has two modes over the same pattern engine:

- `perform`: timed playback of a short suite for demonstration
- `hold_probe`: deterministic frozen states for oscilloscope, digital capture, and host analysis

## Why This Exists

The current repository already has:

- a working CRT signal engine
- scanline hooks
- a framebuffer path
- a calibration-oriented default pattern in `main/app_main.c`

What it does not yet have is a first-class authored showcase that can be defended as both:

- a piece of signal art
- a measurement instrument

This design creates that bridge without prematurely building the entire compositor stack.

## Primary Thesis

The first suite should demonstrate that deterministic composite-video patterns generated by the ESP32 can be used to extract semi-precise electrical response characteristics from the output stage.

Primary result:

- transfer curve estimation for the generated tonal response

Derived results:

- local linearity / contrast compression by tonal region
- overshoot, undershoot, and blooming-adjacent contamination from strong transitions

## Scope

### In scope

- a short authored suite with a fixed movement structure
- reuse of the existing framebuffer and scanline-hook path
- deterministic presets for each movement
- mode switching between `perform` and `hold_probe`
- metadata needed for analysis tooling
- electrical-first measurement workflow
- camera documentation workflow as a secondary instrument

### Out of scope

- a full compositor or scene graph
- sprites, tilemaps, text planes, and authored scene packages
- dynamic asset loading
- general-purpose effect scheduling
- changes to `crt_hal`, DMA scheduling, or sync/burst ownership
- ML or predictive models in the realtime path

## Integration Surface

The suite should remain additive to the existing codebase.

Use existing components:

- `crt_core`
  owns blanking, sync, burst, prep scheduling, and hook invocation
- `crt_fb`
  provides the easiest first surface for deterministic authored patterns
- host analysis tools under `tools/analysis/`
  remain the initial destination for extracted measurements and reports

The suite should attach through the existing scanline hook path first. It must not require rewriting the core stage scheduler in this milestone.

## Runtime Shape

Recommended initial module split:

- `components/signal_suite/`
  movement scheduler, mode selection, state machine, and public control API
- `components/signal_suite_patterns/`
  pattern generators for staircases, ramps, and transition stress states
- `components/signal_suite_markers/`
  compact metadata and preset descriptors exported to host-side tooling

This split is guidance, not a hard architectural requirement. The first implementation may collapse modules if that keeps the milestone smaller without losing interface clarity.

Recommended first integration:

1. initialize a `crt_fb_surface_t` as today
2. generate suite content into the logical framebuffer surface
3. register a suite-owned scanline hook
4. expose the current movement and preset to host tools

This milestone is explicitly locked to `crt_fb` + scanline hook.

A future milestone may introduce a line-scratch renderer or direct active-window generator, but that is deferred until after the first suite is working and measurable.

This keeps the implementation close to what already works while allowing the suite to become its own runtime layer.

## Visual Direction

The suite should follow **Measurement Brutalism**:

- black or near-black fields
- stark ramps and step edges
- sparse use of labels or overlays, if any
- no retro-console mimicry in the first milestone
- no decorative nostalgia layer that obscures the protocol

The performance should read as:

- laboratory display
- instrument panel
- controlled signal event

not:

- fake boot screen
- game intro
- generic synthwave CRT homage

## Suite Structure

Recommended movement set:

1. `lock`
   blank-to-useful onset, alignment, and establishment of signal authority
2. `staircase`
   amplitude steps as the main transfer-curve movement
3. `ramp`
   continuous tone and windowed tone sweeps for local linearity analysis
4. `transition_stress`
   hard edges and alternating regions for overshoot and contamination analysis

These movements should be sufficient for the first milestone. Do not add a fifth movement unless one of the four proves unable to carry the required measurement goals.

## Modes

### `perform`

Purpose:

- public showcase
- visual communication of the suite
- timed authored sequence

Behavior:

- fixed ordering of movements
- fixed or narrowly bounded dwell times
- controlled transitions such as cut, sweep, pulse, or hard hold
- loopable without changing the underlying semantics of each movement

### `hold_probe`

Purpose:

- repeatable measurement
- host-side capture and analysis
- laboratory use

Behavior:

- each movement exposes deterministic named presets
- each preset is stable until explicitly changed
- no hidden modulation while frozen

Example preset families:

- `lock_baseline`
- `staircase_8`
- `staircase_16`
- `ramp_full`
- `ramp_window_low`
- `ramp_window_mid`
- `ramp_window_high`
- `transition_255_0`
- `transition_192_0`
- `transition_checker_1`

## Consistency Rules

The suite must obey these rules:

1. every visual state reachable in `perform` must map to a reproducible semantic state in `hold_probe`
2. `perform` may interpolate between presets, but it must not invent measurement states with no stable equivalent
3. movement metadata must be visible to host tools
4. changing mode must not change the meaning of a given preset

These rules are what make the suite simultaneously artwork and protocol.

## Signal Strategy

The first suite should be **luma-dominant**.

Reasoning:

- the primary thesis is tonal electrical response
- luma-only or luma-dominant states are easier to defend electrically
- color can return later once the transfer and transition workflows are stable

Recommendation for milestone 1:

- keep all primary measurement presets luma-only
- keep `perform` luma-only as well in the initial implementation
- defer color to a later milestone once tonal and transition workflows are stable

## Measurement Strategy

Primary truth source:

- electrical measurement at the ESP32/DAC output stage

Primary instruments:

- oscilloscope
- digital capture device / external ADC workflow

Secondary instrument:

- Logitech C270 camera path already assumed by current analysis tooling

The camera is useful for:

- documentation
- framing verification
- optical comparison
- future closed-loop research

It is not the truth source for this first milestone.

## Camera Assumptions

The repository already assumes a Logitech C270-class capture workflow:

- 1280x720 default capture path in the monitor tool
- 30 fps assumptions in analysis scripts
- webcam-oriented analyzer naming and heuristics

This suite should therefore document the C270 as a characterized secondary sensor with known limits:

- fixed focus
- 720p at 30 fps
- internal processing and MJPEG compression
- unsuitable as the primary source of temporal truth

## Experimental Protocol

Each run should capture:

- video standard
- suite mode
- movement name
- preset name
- whether color is enabled
- oscilloscope scale and sample configuration
- capture hardware metadata
- timestamp and run ID

Core protocol steps:

1. flash and start the suite
2. select `hold_probe`
3. cycle a fixed preset list
4. acquire oscilloscope traces and digital captures
5. store run metadata alongside captures
6. generate host-side plots and derived metrics
7. optionally run the same movement in `perform` for public-facing documentation

## Outputs

This milestone should produce:

- a running suite on ESP32 hardware
- a small public-facing loop in `perform`
- deterministic preset control in `hold_probe`
- exported movement/preset metadata for host tools
- at least one transfer-curve report
- at least one overshoot / transition report
- usage documentation for laboratory operation

## Public API Shape

The initial public control surface should stay small. A likely API shape is:

- initialize suite runtime
- select mode
- select movement
- select preset
- advance or reset perform playback
- query current suite state

Do not design a broad scene-authoring API in this milestone.

## Control and Metadata Contract

Milestone 1 needs a deliberately narrow control plane.

Recommended contract:

- runtime control through a small UART / serial command surface exposed through the existing ESP-IDF monitor workflow
- no network dashboard integration in this milestone
- no binary RPC protocol in this milestone

Required commands:

- select mode: `perform` or `hold_probe`
- select movement
- select preset
- query current state
- advance to next movement or reset the suite loop

Recommended metadata contract:

- firmware exposes a compact static table of movement IDs and preset IDs
- current state is printable over serial in a machine-readable text form
- host capture runs record that serial state alongside oscilloscope and digital-capture artifacts

Example state line:

- `SUITE_STATE mode=hold_probe movement=staircase preset=staircase_16 frame=42 color=off standard=ntsc`

This is sufficient for planning because it bounds the first implementation to:

- a small command parser
- a small state-report formatter
- no dashboard or monitor-tool coupling

## Risks

### Scope drift

Risk:

- turning the suite into the first half of a full compositor

Mitigation:

- keep the runtime narrowly scoped to movements, presets, and measurement metadata

### Decorative drift

Risk:

- adding retro UI layers that obscure the protocol

Mitigation:

- require every visual element in the first suite to justify either a measurement role or a performance role

### Instrument confusion

Risk:

- treating camera output as equivalent to electrical truth

Mitigation:

- keep electrical capture primary in spec, docs, and reports

## Success Criteria

The design is successful when:

- the suite is visually legible as a coherent signal-art piece
- the same runtime produces deterministic probe states
- host tooling can recover a useful transfer curve from captured electrical data
- local linearity and overshoot metrics can be derived from the defined presets
- the implementation remains clearly smaller than the future compositor effort

## Roadmap Position

This suite is the milestone-1 bridge:

- `milestone 1`: authored measurement suite
- `milestone 2`: general-purpose composer / PhosphorOS runtime
- `milestone 3`: more experimental demo-scene and feedback-loop work

The suite should therefore be designed for reuse, but not over-generalized in anticipation of future work.
